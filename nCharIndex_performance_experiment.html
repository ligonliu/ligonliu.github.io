nCharIndex performance experiment

comparison of query and build time between nCharIndex and non-indexed MySQL nCharIndex performance analysis

Query time analysis¶

Read in the data. The data is generated from a benchmark dataset of movie information texts, with common English words as search patterns.

In [1]:
import pandas as pd
df = pd.read_csv("log.csv", names=['pattern','count','no_index_time','index_time'])

  • "pattern" field is the substring used in SQL like queries: "select ... from TABLE1 where textfield like '%{pattern}%'
  • count is the number of rows in the query result.
  • no_index_time is the query time in milliseconds by the MySQL RDBMS without nCharIndex.
  • index_time is the query time in milliseconds with help from nCharIndex.

Observations¶

  • no_index_time is almost the same around 10000ms, regardless of pattern length and count.
  • index_time is positively correlated with count.

Performance advantage¶

In [3]:
acceleration_ratio = df['no_index_time'] / df['index_time']
In [4]:
acceleration_ratio.median(), acceleration_ratio.mean()
Out[4]:
(169.5717893217893, 292.75979690199233)

In average, querying with help of nCharIndex has 292x performance gain over the original RDBMS. The gain is quite prevalent, as the median gain is 169x.

Conclusion¶

nCharIndex can accelerate substring queries by hundreds fold. The acceleration is more significant on substrings that occur less frequently in the database.

The smaller the result set is, the shorter time an indexed query will take. The relation between query time and query result size is highly linear with a relatively small deadweight.

Comparing persistence backends, metall is about 3 times faster than rocksdb.

Index building time is roughly O(l0.65n1.04), where l is the average length of each text item, and n is the number of rows.
